## Part 2: CRITICAL Implementation Issues (Current State)

### 2.1 Auto-Expansion Not Working

**SYMPTOM**: Subnav menus don't unfurl when navigating to child pages.

**ROOT CAUSE**: Carbon's `SideNavMenu` component uses `defaultExpanded` prop, which only sets **initial state on mount**. It does NOT react to prop changes.

**CODE LOCATION**: `Header.js:257`
```javascript
<SideNavMenu
  defaultExpanded={menuItem.expanded}  // ❌ WRONG - not reactive!
```

**WHY IT FAILS**:
1. User navigates to `/Storage/ColdStorageMonitoring`
2. `useMenuAutoExpand` hook runs, sets `menuItem.expanded = true`
3. `autoExpandedMenus` state updates → Header re-renders
4. `SideNavMenu` receives new `defaultExpanded={true}` prop
5. **BUT**: Carbon ignores it because component already mounted with `defaultExpanded={false}`
6. Menu stays collapsed

**THE FIX** (2 options):

**Option A: Force Re-Mount via Key** (Quick Fix)
```javascript
<SideNavMenu
  key={`${menuItem.menu.elementId}-${menuItem.expanded}`}  // Force re-mount when expanded changes
  defaultExpanded={menuItem.expanded}
```

**Option B: Use Controlled Component** (Proper Fix - if Carbon supports it)
Check if `SideNavMenu` has an `isExpanded` prop. If so:
```javascript
<SideNavMenu
  isExpanded={menuItem.expanded}  // Controlled component
  onToggle={() => setMenuItemExpanded(...)}
```

### 2.2 Active State Flickering/Inconsistency

**SYMPTOM**: 
- "Cold Storage Monitoring" shows left border (✅ works)
- "Storage Management" doesn't show left border (❌ fails)
- Active state "flickers" briefly then disappears

**ROOT CAUSE**: **Duplicate active state management** creates CSS conflicts.

**PROBLEM ARCHITECTURE**:
```
Level > 0 items with children:
  SideNavMenuItem (isActive prop)
    ├─ adds `.cds--side-nav__link--current` class
    └─ contains <button class="custom-sidenav-button active">
         └─ ALSO tries to add active border!
```

**CSS CONFLICT** (`Style.css:563-564`):
```css
/* This KILLS all active states!
 */
.cds--side-nav__menu-item .cds--side-nav__link {
  background-color: transparent !important;  /* ❌ Overrides active background */
}
```

**WHY "Storage Management" FAILS**:
1. Rendered as `<SideNavMenuItem isActive={true}>`
2. Carbon adds `.cds--side-nav__link--current` class
3. CSS rule `.cds--side-nav__link--current` tries to apply border + background
4. **BUT** `.cds--side-nav__link { background: transparent !important; }` wins
5. Background stays transparent, border doesn't show (or briefly shows then CSS recalc hides it)

**WHY "Cold Storage Monitoring" WORKS SOMETIMES**:
- It uses `<button class="custom-sidenav-button active">` which has higher specificity:
```css
.cds--side-nav__menu-item .custom-sidenav-button.active {
  border-left: 4px solid var(--cds-link-primary) !important;
  background-color: var(--cds-layer-selected-01) !important;
}
```

**THE FIX**:

**Step 1: Remove the CSS that kills active states**
```css
/* DELETE THIS RULE - it's breaking everything */
.cds--side-nav__menu-item .cds--side-nav__link {
  background-color: transparent !important;  /* ❌ DELETE */
}
```

**Step 2: Simplify active state to single source of truth**

Currently you have TWO systems:
1. Carbon's `isActive` prop on `SideNavMenuItem`
2. Custom `className="active"` on buttons inside

**PICK ONE**:

**Option A: Use ONLY Carbon's isActive** (Recommended)
- Remove custom buttons for simple nav items (renderSingleNavButton)
- Only use custom buttons for dropdown toggles
- Let Carbon handle all active states via `isActive` prop

**Option B: Use ONLY custom buttons**
- Remove `isActive` prop from all `SideNavMenuItem` components
- Handle all active states via `.custom-sidenav-button.active`
- More control but more complex

### 2.3 CSS Specificity Issues

**CURRENT CSS HIERARCHY** (Style.css:562-587):
```css
/* Base - KILLS EVERYTHING */
.cds--side-nav__menu-item .cds--side-nav__link {
  background-color: transparent !important;  /* ❌ */
}

/* Active state - OVERRIDDEN BY ABOVE */
.cds--side-nav__menu-item .cds--side-nav__link--current {
  border-left: 4px solid var(--cds-link-primary);
  background-color: var(--cds-layer-selected-01) !important;  /* ⚠ Loses to base rule */
}

/* Custom button - WORKS because higher specificity */
.cds--side-nav__menu-item .custom-sidenav-button.active {
  border-left: 4px solid var(--cds-link-primary) !important;
  background-color: var(--cds-layer-selected-01) !important;  /* ✅ Wins */
}
```

**SPECIFICITY SCORES**:
1. `.cds--side-nav__menu-item .cds--side-nav__link` = **0,2,0** (base)
2. `.cds--side-nav__menu-item .cds--side-nav__link--current` = **0,2,0** (SAME - tie goes to LAST in file)
3. `.cds--side-nav__menu-item .custom-sidenav-button.active` = **0,3,0** (WINS)

**WHY IT FAILS**:
- Base rule and active rule have SAME specificity
- Both have `!important`
- CSS tie-breaker: LAST rule wins
- **BUT**: Base rule comes AFTER active rule in your CSS file (line 563 vs 567)
- So transparent background wins!

**THE FIX**:
```css
/* Option 1: Remove !important from base rule */
.cds--side-nav__menu-item .cds--side-nav__link {
  /* background-color: transparent; */  /* DELETE THIS ENTIRELY */
}

/* Option 2: Increase specificity of active rule */
.cds--side-nav__menu-item .cds--side-nav__link.cds--side-nav__link--current {
  /* Extra .cds--side-nav__link increases specificity to 0,3,0 */
  border-left: 4px solid var(--cds-link-primary);
  background-color: var(--cds-layer-selected-01) !important;
}
```

**RECOMMENDED**: Option 1 (delete the transparent background rule entirely). You added it to "simplify active styles" but it's actually KILLING them.
