# Implementation Plan: Pharmaceuticals Laboratory Workflow

**Branch**: `feat/283-pharmaceuticals-lab` | **Date**: 2025-12-14 | **Spec**:
`specs/283-pharmaceuticals-lab/spec.md`  
**Input**: Feature specification from `/specs/283-pharmaceuticals-lab/spec.md`

## Summary

Deliver a pharmaceuticals laboratory workflow leveraging the existing OGC-51
notebook/page architecture and storage services. Scope covers accession/QC,
processing, assay notebooks with controls/acceptance, aliquot + hierarchical
storage, retrieval/shipping with chain-of-custody, dashboards, and
disposal/archiving. Reuse existing NotebookService/Page templates and Storage
services to minimize new APIs; add only pharma-specific templates, QC
checklists, and dashboards.

## Technical Context

**Language/Version**: Java 21 (backend), React 17 (frontend)  
**Primary Dependencies**: Spring Framework 6.2.x (MVC, Security), Hibernate 6.x,
Liquibase 4.8.0, HAPI FHIR R4 (as needed), SWR/React Router/Formik/Yup, Carbon
Design System v1.15  
**Storage**: PostgreSQL 14+ with Liquibase migrations; FHIR store if external
sync required  
**Testing**: JUnit 4 + Mockito (backend), Cypress + Jest/RTL (frontend)  
**Target Platform**: Tomcat 10 for WAR backend; React SPA served via existing
frontend build  
**Project Type**: Web (backend + frontend)  
**Performance Goals**: Sub-2 minute accession/label per sample; dashboard
freshness ≤15 minutes; retrieval/shipping log latency ≤5 minutes  
**Constraints**: No @Transactional in controllers; no native SQL; Carbon-only
UI; React Intl for all strings; Liquibase for schema changes; config-driven lab
variation  
**Scale/Scope**: Typical lab throughput (order of hundreds of samples/day);
storage hierarchy reuse (room/device/rack/box/position); notebook pages per
assay run

## Constitution Check

_GATE: Must pass before Phase 0 research. Re-check after Phase 1 design._

- Carbon Design System only; no custom CSS frameworks.
- React Intl for all UI strings; provide at least en+fr keys.
- Backend 5-layer pattern; transactions start in services; controllers delegate
  only.
- Liquibase for all schema changes; include rollback.
- jakarta.persistence annotations; no native SQL.
- Tests required (unit/integration/E2E) with >70% coverage for new code.
- Configuration-driven variation for lab-type specifics; no country-specific
  branching.
- Security: RBAC, audit trail (user+timestamp), input validation;
  chain-of-custody logged.
- FHIR/IHE for external integration if exposed.
